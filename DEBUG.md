## 待办

### 需要拿主意的问题

- **本机开发环境下 SoundCloud 搜不出来、也放不了**：这台机器现在联网要走公司代理，但项目代码默认直连，所以只有 SoundCloud 会失败（网易云、QQ 不受影响，因为它俩是浏览器直接连 CDN，不需要经过项目自己的代理）。启动服务时加上环境变量 `CORP_PROXY_HOST=proxy.nioint.com` 就能解决，已经验证过。要不要把这个配置固定下来，还没定。
- **SoundCloud 的歌大部分播完不会存到本地缓存**：网易云、QQ 的歌只要播过一次就会自动缓存到本地，下次播放不用再联网。SoundCloud 不一定——大部分歌默认走的是一种叫 HLS 的流媒体格式，现在的缓存逻辑没覆盖这种格式，只有少数走普通 mp3 格式的 SoundCloud 曲目才会被缓存。要解决的话有两个方向：一是把 SoundCloud 也强制改成走普通 mp3（代价是会重新出现一个已经修好的播放失败问题）；二是单独写一套逻辑把 HLS 格式的歌也存下来（工作量明显更大，还没验证可行性）。两个都还没做，需要再合计一下选哪个。
- **线上偶发 SoundCloud 搜不出来/播放不了**：用户反馈线上 `https://listening-5bnv.onrender.com/` 手机和电脑都遇到过打不开，但排查时问题已经自己好了，没能现场复现。排查时确认了：Render 后端代理本身没问题（`/sc-client-id`、搜索接口当时都能正常拿到数据，响应 1-3 秒），线上部署的代码也和仓库源码一致，不是跑着旧版本。所以问题大概率出在用户设备到 SoundCloud 之间，而不是服务端。怀疑方向（都还没验证）：
  1. 播放走 HLS 格式时，音频分片是浏览器直接连 SoundCloud 的 CDN（`playback.media-streaming.soundcloud.cloud`），不经过 Render 转发——如果这个域名在用户当时所在的网络下不通，会导致播放失败，但不影响搜索。
  2. Render 免费实例冷启动：前端探测代理是否可用的超时只有 6 秒，如果刚好撞上实例刚从休眠唤醒、响应变慢，会误判代理不可用，退化成浏览器直连 SoundCloud（这条路在国内是不通的）。
  3. 浏览器扩展（广告拦截等）拦截了带 soundcloud 相关关键词的请求。
  下次再遇到，优先让用户看浏览器 Console / Network 面板的报错信息，能更快定位卡在哪一步。

**2026-07-11**
- 修了"点一首歌会把播放队列顺序打乱"的问题：队列面板、歌单详情页、搜索结果里点一首歌播放，走的都是
  同一个 `playTrackAt(id)`，原来的逻辑不管这首歌在不在播放队列（`playOrder`）里，一律把它插到队列
  最前面（先从原队列过滤掉再插进去）。改成先判断这首歌是否已经在 `playOrder` 里：在的话直接播放、
  不改变队列顺序；不在的话才插到最前面。用 Playwright 预置了一份带播放队列和歌单的本地存储数据，
  验证过队列面板点队列中间的歌、歌单里点已在队列中的歌、歌单里点不在队列中的歌这三种场景，行为都符合
  预期；"播放全部"整体替换播放队列的行为不受影响（不是这次改的函数）。
- 修了"点一首歌显示播放失败，反复点同一首还是失败"的问题：`_onError`（音频元素 error 事件）、HLS
  播放的 fatal error、`play()` 被拒绝、`ensureTrackDetails` 失败这四个失败点原来都是直接弹一次
  "播放失败"就完了，没有任何重试或者刷新播放链接的逻辑。根因跟音源有关：QQ音乐/SoundCloud
  （`src/api/qq.js`/`src/api/soundcloud.js`）拿到的是带时效性 token 的源站直链，`track` 对象持久化
  到 `localStorage` 后过期链接会一直带着，之前的代码只要 `track.audioUrl` 有值就直接用、不重新解析，
  导致每次点都用同一个早就失效的链接必然失败；网易云（`src/api/netease.js`）走的是第三方公共代理
  `api.qijieya.cn`，本身没有个人时效性 token，失败更多是这个第三方服务临时不稳定，或者歌曲本身
  VIP/付费限制根本没有可用链接。新增 `handlePlaybackFailure(reason, myToken)` 作为四处失败的统一
  入口：失败后清空这首歌已经解析过的 `audioUrl`/`detailsLoaded`/`scIsHLS`，逼 `loadAndPlay` 重新走
  一遍 `ensureTrackDetails` 问源站要新链接，重试一次；重试过还失败才真正提示"播放失败"，避免无限重试。
  开发过程中用 Playwright 造了个必定 404 的假播放链接实测，发现并修掉了一个真实 bug：一次真实失败会
  同时触发音频 `error` 事件和 `play()` 被拒绝两个独立信号，不去重的话会被当成两次失败分别处理，导致
  重试没完没了（实测跑到 194 次才手动掐断）；改成给每次 `loadAndPlay` 调用编号（`this._loadToken`），
  失败信号带着编号，跟当前最新编号对不上或者这次编号已经处理过的信号直接丢弃，才把重试次数锁定在
  精确的 1 次。这个改动不影响本地音频缓存（`audioCacheKey` 只依赖音源自己的稳定 id，跟这次清空的
  字段无关，缓存命中判断照常）。
- 真机复测确认了"锁屏后台自动切歌"问题的真正根因：不是代码里 Media Session 状态管理的问题（虽然那个也是真实存在的小 bug，已经修了），而是**手机锁屏后 Android 系统的电池优化策略直接限制了 Chrome 的后台执行**——用户手动在系统设置里给 Chrome"允许后台耗电"之后，问题彻底消失；不锁屏、只是切到别的 App 用（屏幕亮着）则可以无限稳定连续切歌，印证了这一点。这一层限制网页代码没有办法自己绕过去（没有 API 能让网页自己申请"忽略电池优化"，这是平台故意设的安全边界，只有真正的原生 App 用前台服务才能天然免检，工作量和不确定性都大得多，暂不考虑）。这次改动做的是力所能及的缓解：加了 PWA 安装支持（`examples/manifest.json` + `examples/icons/` 两张图标 + `<head>` 里补的 `<title>`/`manifest`/图标相关标签 + `server.mjs` 的 MIME 表加了 `.png`），装上之后在系统电池管理列表里会是一个独立好认的 App（而不是笼统的"Chrome"），方便用户自己找到并单独放行；同时加了一条一次性引导提示（安卓 + 移动端窄屏时出现，点"知道了"记住不再提示，`localStorage` key 是 `lsn-battery-hint-dismissed`），直接告诉用户去系统设置里放行。图标是用 Playwright 截图生成的，复用了页面已有的紫色渐变 + 白色音符图案，没有引入新的设计语言。特意确认过 `sw.js` 没有改动、也没有让它开始缓存页面 HTML/JS——`server.mjs` 的静态文件路由本来就不设任何 `Cache-Control`，改完 HTML 重新部署之后用户下次打开就是最新版本，不需要额外的"应用更新"步骤，只有 manifest 里图标/名字这类外壳信息 Chrome 会隔几天才刷新一次，不影响日常改动。用 Playwright 端到端验证过 manifest/图标能正确加载、引导提示在安卓窄屏下正确出现、点掉之后刷新页面不再出现、桌面/非安卓下不出现，桌面三栏布局没受影响。真机上"添加到主屏幕"效果如何、装上之后电池管理列表里能不能找到独立条目，还是要用户自己确认。
- 修了 2026-07-10 那次 Media Session 修复（commit dcad4a4）自己留的一个漏洞："手机后台自然切歌"实测下来是第一首播完能切到第二首，但通知栏在那一刻就消失了，再也切不到第三首。根因：一首歌自然播完时浏览器会先触发一次真实的 `pause` 事件（`paused` 变 true 是"播完了"这套流程自带的步骤，不是用户暂停），而 `_onPause` 不分青红皂白地把这次假暂停也同步成了 `navigator.mediaSession.playbackState = 'paused'`；下一首真正开始播放前还有一段异步过程（读本地缓存/联网取播放详情），这段时间里系统看到的就是"已暂停"，很可能因此收起通知、降低这个页面的后台执行优先级，导致下一次自动切歌需要的异步操作被拖住，永远等不到触发。修法是让 `_onPause` 在 `a.ended` 为 true 时直接跳过（这种情况下一定是紧接着要切下一首，不是真暂停），真正的用户暂停不受影响。这台环境仍然没法模拟真机后台限流场景，改动本身没法在这里验证效果，需要部署后在真机后台场景下实测确认。
- 修了"自然切歌到没缓存过的歌时不顺利"的问题：根因是三个叠加的问题——完全没有预加载（切歌那一刻才从零开始联网解析下一首）、加载过程中界面没有任何反馈（唱盘照转、按钮显示暂停图标，实际无声，容易让人觉得卡住了）、加载失败会静默卡死（各音源解析函数会吞掉自己的错误，永远不会自动恢复，网络不好时最容易撞见）。加了三块：歌曲快放完时提前在后台解析下一首的播放链接并下载缓存（列表循环、随机模式都覆盖，随机模式是把"掷骰子"提前到快放完那几秒定下来，保证预取的和真正播的是同一首）；播放按钮加了局部的转圈加载提示（不是恢复之前删掉的全局 toast）；加载超时会自动当成"播完了"处理并跳下一首，连续失败超过 2 次就停下来，避免离线时把整条队列跳穿。用 Playwright 在真实浏览器里端到端验证了预取时机/去重、随机模式的锁定机制、加载图标的各种状态切换、卡死自动恢复和熔断，都符合预期；但"真手机、真实慢网络下顺不顺"这个主观感受还是要部署后自己试。
- 修了"喜欢的音乐"顺序不对的问题：点喜欢之后新歌应该排最前面，但原来"喜欢的音乐"的显示顺序直接用 `s.liked`（只是个 id -> boolean 的集合，没有顺序概念）的 key 顺序，JS 里对象的整数型 key 天生按数值升序遍历，跟点喜欢的先后完全对不上；而且列表本身还错误地按曲库（`s.queue`）的顺序取歌，不是歌单自己的顺序。新增 `likedOrder` 字段专门记录点喜欢的先后（最新的排最前面），顺手把歌单详情页的列表显示也改成统一按各自的 `trackIds`/`likedOrder` 取歌，不再依赖曲库顺序（这个顺带修的问题以前对自建歌单也有影响，只是不容易踩到）。用 Playwright 端到端验证过：新喜欢一首歌后确实排到了"喜欢的音乐"第一位，本地持久化的顺序也是对的。
- 修了本地音频缓存的 key 选错字段的问题：`audioCacheKey` 原来用的是 `t.id`（`normalizeTrack` 每次分配的临时内部 id），同一首歌只要重新搜索或重新导入一次，内部 id 就会变，导致明明本地已经缓存过音频文件，却因为 key 变了查不到，白白重新联网下载一遍。改成用音源自己的真实曲目 id（`songid`/QQ 的 `qqId`/`songMid`），同一首歌不管内部 id 怎么变，缓存 key 都稳定。是做"导出歌单/覆盖导入"时顺带发现的（覆盖导入会让整个歌单的歌都拿到新内部 id，这个问题原来就有，但这么改之后更容易踩到）。
- 加了"导出歌单"：歌单详情页跟"播放全部"同一排新增按钮，点一下把这个歌单编码成一段字符串（`LSNPL1:` 开头，内容是精简 JSON 的 base64）复制到剪贴板，按钮文字临时变成"已导出到剪贴板"。"导入歌单"对话框相应支持粘贴这段字符串（原来只认网易云分享链接）。
- 顺带把导入行为从"新建/叠加"改成"覆盖"：填了歌单名且跟已有非内置歌单同名，会直接替换那个歌单的内容而不是再建一个重名的；名字留空则把"喜欢的音乐"整个替换成这次导入的内容（不是合并，之前喜欢但不在这次导入里的歌会被摘掉——这是跟用户确认过的有意行为，对话框小字里也写清楚了）。用 Playwright 在真实浏览器里跑通了导出（含真实剪贴板 API 读写）和两种覆盖场景的验证。
- 修了"手机浏览器切到后台、当前歌曲播完不会自动切下一首"：根因是代码里从没注册过 Media Session（`navigator.mediaSession`），移动端浏览器没有"这个标签页在持续播放媒体"的系统级信号，切后台后可能限流/拦截 `ended` 事件里异步发起的下一首播放请求。加上了播放控制的 Media Session 注册（连带获得锁屏/通知栏的媒体控件）、每次换曲目同步歌曲元数据，顺手去掉了切歌路径上一个多余的 `setTimeout(0)` 延迟。这台环境没法模拟手机切后台的限流场景，只做了本地静态检查和桌面浏览器跑起来的基本验证，真实效果还需要在手机上背景播放实测确认。

**几个容易忘记的背景知识**

- SoundCloud 播放优先走 HLS 格式，是为了绕开另一种格式（progressive mp3）在部分曲目上会被 CDN 拒绝访问的问题。
- 本机如果所在网络需要走公司代理才能联网，启动服务要加 `CORP_PROXY_HOST=proxy.nioint.com` 环境变量，SoundCloud 相关功能才能用；线上部署（Render）不需要这个配置。
- 列表/网格视图切换、酷我音源、JOOX 音源，都是评估过后用户决定不做/下线的，不是遗漏。
